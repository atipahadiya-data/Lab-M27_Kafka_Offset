# StreamPulse Kafka Failure Scenarios with Example Logs
```

| Scenario | Detection | Example Logs | Impact | Response |
|----------|-----------|-------------|--------|---------|
| Single Broker Down | Broker container stopped (`docker ps`), producer/consumer logs show connection refused or reset | `%3|...| Connect to ipv4#127.0.0.1:9094 failed`<br>`%6|...| Disconnected: connection reset by peer` | Writes paused if leader affected; Reads may disconnect; Low data loss risk if ISR maintained | 1. Identify broker: `docker ps | grep kafka`<br>2. Start broker: `docker start <broker>`<br>3. Monitor logs for leader election<br>4. Ensure consumer reconnects |
| Leader Crash / Election | Consumer/producers log disconnections; `kafka-topics --describe` shows leader change for partition | `%6|...| Disconnected: connection closed by peer`<br>`Topic ft-streampulse-events partition count changed from 6 to 0` | Writes paused for affected partitions; Reads temporarily unavailable | 1. Identify leader for partition: `docker exec kafka-1 kafka-topics --describe --bootstrap-server kafka1:9092 --topic ft-streampulse-events | head -3`<br>2. Stop leader broker: `docker stop kafka-<leader-id>`<br>3. Observe leader election<br>4. Restart broker |
| min.insync.replicas Violation | Producer logs: `NotEnoughReplicasException`, `Local: Message timed out` | `[60s] Produced=600, Delivered=0, Failed=0`<br>`❌ FAILED seq: 4 — KafkaError{code=_MSG_TIMED_OUT}` | Writes fail until sufficient replicas online; Reads possible if leader available; Data loss risk if messages not acknowledged | 1. Stop 2 brokers: `docker stop kafka-2 kafka-3`<br>2. Monitor producer failures<br>3. Bring one broker back: `docker start kafka-2`<br>4. Verify writes resume |
| Consumer Crash / Rebalance | Consumer logs: `Consumer group session timed out`, `revoking assignment and rejoining group` | `%4|...| SESSTMOUT: Consumer group session timed out`<br>`Killed: 9` | Reads temporarily unavailable; possible duplicates | 1. Find consumer PID: `ps aux | grep ft_consumer`<br>2. Hard kill: `kill -9 <PID>`<br>3. Observe rebalance logs<br>4. Start new consumer<br>5. Verify lag recovery |
| Rolling Broker Restart | Planned maintenance; monitor logs | `%3|...| Connect to ipv4#127.0.0.1:9093 failed`<br>`%6|...| Disconnected: connection reset by peer` | Writes may temporarily fail if leader restarts; Reads may disconnect briefly | 1. Restart brokers one at a time:<br>`docker restart kafka-1 && sleep 30`<br>`docker restart kafka-2 && sleep 30`<br>`docker restart kafka-3 && sleep 30`<br>2. Monitor logs for zero downtime<br>3. Verify ISRs restored and consumer lag normal |

```
